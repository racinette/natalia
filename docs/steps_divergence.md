# Steps — divergence (docs/ ↔ REFACTOR.MD ↔ src/types)

> Parked working note. Goal: decide the canonical API for steps, then reconcile all three sources. No source is authoritative yet.

## Three-source status
- **docs/steps.md**: Thorough, current-shaped narrative — `defineStep({ args, result, execute, retryPolicy, compensation })`, definition-vs-call-site retry/timeout, and a deep compensation/`undo` model. Mostly matches code; it just *describes the full intended undo surface* (topics/queues/attributes/external from undo) that the impl has not wired yet.
- **REFACTOR.MD Parts 1/2/5/18**: The detailed spec the code was built from. Part 1 (execution/entry model), Part 2 (compensation-on-the-definition), Part 5 (`RetryPolicyOptions` + `StepBoundary` call-time `timeout`), Part 18 (schema-backed `args`/`result`). Code tracks it closely. One naming gap (`StepCompensationDefinition` vs Part 2's `StepCompensation`) and the per-instance-primitive slot list is broader in Part 2/docs than wired.
- **src/types (impl)**: Implemented for the core: `defineStep`, `StepDefinition`, `StepCallOptions`/`StepTimeoutCallOptions`/`StepBoundary`, `RetryPolicyOptions`, `CompensationInfo`, `StepCompensationDefinition`, three-overload `StepAccessor`, branded `CompensationId`. **Partial**: compensation `undo` ctx only wires `steps`/`requests`/`queues(=never)`/`children`/`external` + base-context `channels`/`streams`/`events`; no `topics` accessor and no per-instance `attributes` accessor on `CompensationContext` (steps.ts comment: "accessors land in their own steps 13/15/16"). Regression tests `02`/`03`/`08` pass and exercise the implemented surface.

For the steps primitive itself (retry/timeout/args/result/entry model), all three barely diverge — they are essentially aligned. The real drift is concentrated in the compensation `undo` accessor surface (docs/Part 2 ahead of code) and minor naming.

Confirmation re: the cross-cutting cleanup — for **steps**, call-site `retry` (`StepCallOptions.retry`) and the `StepBoundary` `timeout` are **legitimate and present in all three** (steps are retried; `retry`-only overload exists, test `03` lines 115-135). This is the opposite of child workflows, where `retry` was wrongly copied on; test `03` lines 206-213 confirms children reject a `retry`-only options bag.
